# Jenkins 与阿里弹性容器发布 Skill

## 1. 适用场景

用于每个版本开发完成后，检查测试环境和正式环境发布能力。Jenkins 负责本地构建、自动测试和测试环境发布；阿里弹性容器负责正式环境部署和正式版本演示。

## 2. 输入

- 当前 milestone 文件。
- 当前服务或前端项目目录。
- Maven、npm、Python 构建命令。
- Dockerfile 或镜像构建配置。
- Jenkins 流水线配置。
- 阿里弹性容器正式环境配置。

## 3. 执行步骤

1. 确认当前版本涉及的前端和后端模块。
2. 编写或更新 Jenkins 流水线。
3. Jenkins 执行依赖安装、单元测试、前端构建、AI 测试。
4. Jenkins 构建测试镜像或测试构建产物。
5. Jenkins 发布到本地 Docker / Docker Compose 测试环境。
6. 检查阿里弹性容器正式环境配置。
7. 正式环境使用 release 制品和正式镜像。
8. 检查环境变量、密钥、健康检查、日志和回滚说明。
9. 将测试发布结果和正式发布检查写入 milestone 实现记录。

## 4. 检查清单

- [ ] Jenkins 凭证未写入仓库。
- [ ] 测试环境使用 snapshot 制品。
- [ ] 正式环境使用 release 制品。
- [ ] 镜像不包含 `.npmrc`、token、构建缓存和明文密钥。
- [ ] 前端、后端、AI 构建命令都在流水线中体现。
- [ ] 测试环境发布后可访问健康检查或页面。
- [ ] 阿里弹性容器正式环境配置包含环境变量和密钥管理。
- [ ] 正式环境有健康检查和回滚说明。

## 5. 推荐提示词

```text
请使用 docs/plan/skill/cicd-jenkins-eci-skill.md，为当前 milestone 补充 Jenkins 测试环境发布和阿里弹性容器正式环境发布检查。完成后把结果写入 milestone、handle 和 PROGRESS.md。
```

## 6. 常见坑

1. 把部署留到最后一个版本才处理。
2. Jenkins 只构建后端，漏掉当前模块前端。
3. 测试版本误用 release 制品。
4. 正式镜像带入 `.npmrc` 或 token。
5. 正式环境没有健康检查，部署后无法判断是否可用。
