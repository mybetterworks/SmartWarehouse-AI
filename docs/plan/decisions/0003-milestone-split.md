# 0003 敏捷版本拆分方案

## 状态

ACCEPTED

## 背景

项目需要通过 vibe coding 持续开发，并在每个版本后形成可演示或可运行成果。版本数量不能太多，以免管理复杂；也不能太少，否则每版改动过大，不利于学习、验证和复盘。

版本拆分已经根据后续确认的商用协作节奏调整为：先甲方组件库二次开发，再甲方前后端基座，再乙方各自前后端项目，最后进行多乙方交互业务和 AI 业务联动发布加固。

## 决策

当前版本拆分为 8 个 milestone：

| 版本 | 目标 |
|---|---|
| V01 | 甲方组件库二次开发与最小项目骨架 |
| V02 | 甲方前后端基座与 CI/CD 基线 |
| V03 | WMS 乙方前后端基础模块 |
| V04 | MES 乙方前后端基础模块 |
| V05 | AI 乙方前后端基础模块 |
| V06 | 甲方 task 运营统计前后端模块 |
| V07 | MES、WMS、task 多方交互业务 |
| V08 | AI 与业务服务交互及发布加固 |

## 影响

1. `PROGRESS.md` 按当前 8 个版本管理进度。
2. `milestones`、`study`、`handle` 都按当前 8 个版本维护对应文件。
3. 每个版本完成后必须可演示或可运行。
4. 每个业务模块版本必须同时覆盖后端、前端、接口联调、Jenkins 测试发布和阿里弹性容器正式发布检查。
5. milestone 文件已经承载对应版本所需的架构、业务、接口、数据库和验收描述，后续开发具体版本时优先读取 milestone，不重复读取 `docs/design`。

## 后续动作

1. 如果某个版本过大，可以新增子版本或拆分新 milestone。
2. 如果版本拆分再次发生重大调整，本文件改为 `SUPERSEDED`，并新增后续决策文件。
